### [SEGALL: A Unified Active Learning Framework for Wireless Sensing Data Segmentation](https://dl.acm.org/doi/pdf/10.1145/3749492)

1. 论文最强的证据不是单一准确率，而是同一架构跨 IMU、Wi-Fi、mmWave 三种复杂度差异显著的模态仍保持约 91%-95% 的分割准确率。

   IMU 通常只有 x、y、z 三维，信号较稳定。

   Wi-Fi 使用 CSI/DFS，受多径和环境变化影响。

   mmWave DFS 可达约 128 维，空间分辨率和数据复杂度更高。
   普通双层 LSTM 在三种信号上分别只有 82.17%、12.33% 和 14.30%，说明复杂模态不是简单模型就能处理的。SegALL 在三者上都保持高准确率，因此其 Transformer 全局建模、LSTM 局部平滑、二分类分割的组合具有一定的跨模态适用性。

2. 主动学习的关键创新是把“序列输出的方差”与“0-1 转换次数”结合；它是否能泛化到目标密度和干扰结构完全不同的数据，仍需外部数据验证。

   SegALL 将分割输出表示为 0-1 序列，并定义：

   $[ Q=\sigma^2\left(1+\frac{M}{N-1}\right) ]$

   其中，$(\sigma^2) $是序列方差，$(M)$ 是相邻位置发生 0↔1 转换的次数。它的直觉是：

   - 方差高：输出中的 0 和 1 混杂，模型不像是明确判定为全背景或稳定目标。
   - 转换次数多：预测片段破碎，模型频繁在“目标”和“非目标”之间摇摆。
   - 两者同时高：该序列可能包含模型难以区分的干扰，值得优先标注。

   这个设计的优点是计算便宜、适合长序列，而且比只看单个样本预测概率更贴合“分割”任务。

   但它可能受到几个混淆因素影响：

   - 真正包含很多短目标活动的序列，本来就会有较多 0-1 转换，即使没有干扰，\(Q\) 也可能较高。
   - 目标活动占比会影响二值序列方差。目标密度接近 50% 时方差较高，极度稀疏时反而较低。
   - 转换次数只能反映输出破碎程度，不能判断破碎是由干扰、真实活动密集，还是模型校准不佳造成的。
   - 论文没有明确说明主动学习阶段的二值化阈值是否始终与评估时的 0.7 相同，因此阈值敏感性仍不清楚。

   真正验证泛化性，至少还需要：

   - 按目标密度、目标持续时间和干扰强度分层测试。
   - 检验 \(Q\) 与实际分割误差之间的相关性。
   - 在相同标注预算下，与 entropy、margin、BALD、core-set 等策略比较。
   - 在全新用户、环境、设备和干扰类型上外部验证。

   所以目前可以说它“在本文三组数据上有效”，但不能直接说它是普遍适用的序列主动学习指标。

3. Wi-Fi 数据由已有手势片段合成连续序列，而 mmWave/IMU 为长时真实采集；三类模态的数据生成方式并不完全对等。

   mmWave 和 IMU 是连续真实采集：

   - 6 名志愿者。
   - 办公室、卧室、客厅三个环境。
   - 每次连续采集 30 分钟。
   - 日常活动与随机发生的目标手势共同存在。

   Wi-Fi 则来自 Widar3.0 的已有手势片段，作者把目标手势和其他手势随机拼接为连续序列。

   这种合成方式包含“语义相近的干扰手势”，比简单加入白噪声更合理，但仍可能缺少真实连续采集中的现象：

   - 动作之间自然过渡和共同发力，即 co-articulation。
   - 设备漂移、时间同步误差和背景长期变化。
   - 真实动作边界模糊，而拼接边界可能相对清晰。
   - 用户走动、遮挡、环境改变造成的连续信道变化。
   - 空闲背景与动作片段之间的非平稳过渡。

   因此，三种模态的结果可以支持“SegALL 能够适配不同数据形态”，但不能严格用于比较“哪种信号更难”或者“不同模态提升幅度是否完全公平”。

   更强的实验设计，是让同一批受试者在同一时间、同一环境中同步采集 IMU、Wi-Fi 和 mmWave，使用相同活动序列、边界标注及用户独立划分。

4. 准确率分母是所有检测区间而非所有真实目标区间，还应关注论文未单独报告的漏检率或召回率。

   论文定义：

   $[ Accuracy= \frac{\text{Correctly Detected Segments}} {\text{Total Number of Detected Segments}} ]$

   预测区间只有在 IoU ≥ 0.85 时才算正确；分母包括正确预测、低 IoU 预测和被误判为目标的干扰区间。

   关键问题是：没有被模型检测出来的真实目标区间，并未明确进入分母。因此，该指标在统计意义上更接近：

   > IoU ≥ 0.85 条件下的 segment precision。

   例如，数据中有 100 个真实活动：

   - 模型 A 只预测 50 个，其中 48 个正确，论文指标为 96%。
   - 模型 B 预测出 100 个，其中 90 个正确，论文指标为 90%。

   按论文指标，A 更好；但 A 的召回率只有 48%，B 则可能达到 90%。因此，论文报告的 91%-95% 不能直接解释为“91%-95% 的真实目标都被检测到”。

   还应报告：

   - Segment recall：正确检测数 / 真实目标总数。
   - Precision、recall 和 F1。
   - 漏检数、误检数或每分钟误报次数。
   - 起点和终点的平均边界误差。
   - 不同置信度阈值、不同 IoU 阈值下的 PR 曲线或 AP。

   论文没有单独报告这些指标，所以目前无法判断 SegALL 是否通过少预测一部分困难目标而获得较高分数。

5. 系统尚未在雷达连续流接口上进行端到端在线部署，Raspberry Pi 数值是固定窗口离线推理延迟。

   Table 3 使用一分钟窗口、15 Hz 采样率，即一个窗口约 900 个时间戳。Raspberry Pi 上的推理时间是：

   - IMU：150.21 ms

   - Wi-Fi：463.15 ms

   - mmWave：632.35 ms

     这说明设备处理一分钟数据所需计算时间不到一秒，吞吐量是足够的。

     但端到端实时系统还包括：

     1. 等待数据进入窗口。
     2. 原始 IF/CSI 信号采集。
     3. 校准、降采样和 DFS 计算。
     4. 模型推理。
     5. 二值化、边界合并和下游响应。

     如果必须收满一分钟数据再推理，那么即使模型只计算 632 ms，用户仍可能等待接近一分钟。这是“计算延迟”和“业务响应延迟”的区别。

     此外，原文未明确说明 Table 3 是否包含 DFS 等预处理、设备数据传输和后处理时间。所用 TI 雷达也没有连续流接口，当前流程是在采集结束后处理固定长度数据，而不是雷达端持续在线运行。作者自己把流式处理、重叠窗口、模型压缩和量化列为未来工作。

     因此，更准确的结论是：

     > SegALL 已证明具有边缘设备上的计算可行性，但尚未证明端到端在线部署能力。

     整体看，这篇论文是较强的跨模态方法原型，但还不是完整的部署级验证。最值得后续补强的是统一真实数据采集、完整 precision-recall 评价、外部主动学习验证，以及真正的流式端侧测试。
